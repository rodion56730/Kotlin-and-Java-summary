
# 🧠 Дженерики в Java — Полный конспект для собеседования

---

## 📘 Что такое дженерики

**Дженерики (Generics)** — механизм обобщений в Java, который позволяет писать классы, интерфейсы и методы,  
работающие с разными типами данных **без потери типобезопасности**.

Пример:

```java
Box<String> box = new Box<>();
box.setItem("Hello");
String s = box.getItem();
```

---

## 🧱 Пример базового дженерик-класса

```java
public class Box<T> {
    private T item;

    public void setItem(T item) {
        this.item = item;
    }

    public T getItem() {
        return item;
    }

    @Override
    public String toString() {
        return "Box contains: " + item;
    }
}
```

Использование:

```java
Box<Integer> intBox = new Box<>();
intBox.setItem(42);
Integer num = intBox.getItem();
```

`T` — это **параметр типа** (type parameter), заменяемый на конкретный тип при создании объекта.

---

## 🎯 Зачем нужны дженерики

1. **Типобезопасность во время компиляции** — ошибки ловятся раньше.
    
2. **Безопасное извлечение без кастов** — не нужно `(String)` или `(Integer)`.
    
3. **Универсальный код** — один класс для любых типов.
    
4. **Чистый и понятный API** — `List<String>` ясно говорит, что хранит строки.
    
5. **Совместимость с наследованием через wildcard'ы.**
    
6. **Меньше runtime-ошибок (`ClassCastException`)**.
    

---

## ⚙️ Стирание типов (Type Erasure)

При компиляции информация о типах **удаляется**, и дженерики становятся обычными классами.

Пример:

```java
class Box<T> { T getItem() { ... } }
```

После компиляции:

```java
class Box { Object getItem() { ... } }
```

### Последствия:

- Нельзя создавать `new T()`, `new T[]`.
    
- Нельзя использовать `instanceof T`.
    
- Нельзя наследовать `Exception<T>`.
    
- Возможны **bridge-методы**, чтобы сохранить полиморфизм.
    

---
### 🔹 Bridge-методы

- **Что это:** методы, которые **компилятор создаёт автоматически** при стирании типов.
    
- **Зачем:** чтобы сохранить **полиморфизм** между дженерик-классами.
    
- **Пример:**
    
    `class Parent<T> { T get() { ... } } class Child extends Parent<String> {     String get() { ... }     // Компилятор добавит bridge:     // Object get() { return get(); } }`
    
- **Итог:** Bridge-методы — "мостики", чтобы дженерики работали корректно после type erasure.

---
### 🔹 Raw Type

- **Что это:** дженерик без указания типа (`List list = new ArrayList();`).
    
- **Зачем:** для **обратной совместимости** со старым кодом (до Java 5).
    
- **Минус:** теряется **типобезопасность**, возможен `ClassCastException`.
    
- **Пример:**
    
    `List raw = new ArrayList(); raw.add("Hi"); raw.add(123); // допустимо — опасно`
    
- **Правильно:** `List<String> list = new ArrayList<>();`
---
## 🧩 Наследование дженериков

```java
class Box<T> {}

// 1. Конкретный тип
class PaperBox extends Box<Paper> {}

// 2. С тем же параметром
class GenericBox<T> extends Box<T> {}

// 3. С дополнительными параметрами
class DoubleBox<T, U> extends Box<T> {}
```

### Важно:

- Дженерики **инвариантны**:
    
    ```java
    Box<Paper> ≠ Box<Garbage>
    ```
    
- Массивы, наоборот, **ковариантны** (`Paper[]` можно присвоить `Object[]`).
    

---

## 🔄 Переопределение методов с дженериками

- Сигнатура после стирания типов должна совпадать.
    
- Можно менять имя параметра типа (`<T>` → `<U>`).
    
- Нельзя добавлять новые ограничения (`extends`).
    
- Возвращаемый тип может быть подтипом исходного.
    

Пример bridge-метода:

```java
class Parent<T> { T get() { return null; } }

class Child extends Parent<String> {
    @Override
    String get() { return "ok"; }
    // Компилятор создаёт bridge-метод:
    // Object get() { return get(); }
}
```

## 🧭 Подтип и супертип

|Термин|Значение|Пример|
|---|---|---|
|**Супертип (supertype)**|Родительский тип|`Animal`|
|**Подтип (subtype)**|Дочерний тип|`Dog extends Animal`|

```java
Animal a = new Dog(); // ✅ подтип можно присвоить супертипу
Dog d = (Dog) a;      // ⚠️ обратное требует каст
```

> Подтип наследует поведение супертипа, но не наоборот.

## 🃏 Wildcards (джокеры) в дженериках

`?` — **неизвестный тип**.  
Используется, когда не важен точный параметр типа,  
но важно соблюдать типобезопасность при чтении или записи.

---

### 🔹 Виды wildcard'ов

|Вид|Произносится как|Что делает|Где использовать|
|---|---|---|---|
|`?`|неизвестный тип|ничего нельзя читать или писать (кроме `null`)|редко|
|`? extends T`|“наследует T”|можно **читать** как `T`, нельзя **добавлять**|когда **читаешь**|
|`? super T`|“родитель T”|можно **добавлять T**, нельзя **читать как T**|когда **записываешь**|

---

## 🧠 Принцип PECS — **Producer Extends, Consumer Super**

|Контейнер|Роль|Можно|Нельзя|
|---|---|---|---|
|`? extends T`|Producer (источник)|Читать как `T`|Добавлять|
|`? super T`|Consumer (приёмник)|Добавлять `T`|Читать как `T`|

🧩 **P**roducer → **E**xtends  
🧩 **C**onsumer → **S**uper

---

### Пример: `? extends T`

```java
List<? extends Number> numbers = List.of(1, 2, 3);

Number n = numbers.get(0); // ✅ можно читать
numbers.add(10);           // ❌ нельзя добавлять
```

Компилятор не знает, `List<? extends Number>` — это `List<Integer>` или `List<Double>`.  
Добавление нарушило бы типобезопасность.

---

### Пример: `? super T`

```java
List<? super Integer> list = new ArrayList<Number>();

list.add(1); // ✅ можно писать
Object o = list.get(0); // ✅ читается как Object
Integer i = list.get(0); // ❌ нельзя читать как Integer
```

Компилятор не знает, какой именно супертип (`Number`, `Object` и т.п.),  
поэтому при чтении он “опускает” до `Object`.

---

### Пример: комбинированный метод (копирование)

```java
static <T> void copy(List<? super T> dest, List<? extends T> src) {
    for (T item : src) {
        dest.add(item);
    }
}
```

✅ `src` — Producer → `extends`  
✅ `dest` — Consumer → `super`

---

## 🧩 Когда какой wildcard использовать

|Ситуация|Wildcard|Почему|
|---|---|---|
|Нужно только читать|`? extends T`|Безопасно извлекать|
|Нужно только писать|`? super T`|Безопасно добавлять|
|Тип не важен вообще|`?`|Минимум информации|
|Нужен гибкий API|`extends` и `super` вместе|Универсально|

---

## ⚠️ Частые ошибки

1. `List<Integer>` ≠ `List<Number>`
    
2. Путаница между `extends` и `super`
    
3. Пустой `?` без причины
    
4. Попытка писать в `? extends`
    
5. Попытка читать из `? super`

## 💡 Практические советы

- ⚠️ Избегай **raw types** — они ломают типобезопасность.
    
- ✅ Используй `extends` / `super` в API — гибко и безопасно.
    
- 🧩 Проверяй типы при компиляции, а не в runtime.
    
- 🧱 Для коллекций:
    
    - `List<? extends T>` → безопасно читать.
        
    - `List<? super T>` → безопасно писать.
        
- 📦 Дженерики — это **инструмент проектирования API**, а не просто синтаксис.
    

---

## 🧾 Резюме

> **Дженерики = типобезопасность + переиспользование + гибкость.**

- Они устраняют `ClassCastException`.
    
- Делают код читаемым и универсальным.
    
- Работают через **type erasure**.
    
- Позволяют проектировать гибкие интерфейсы через **wildcards и PECS**.
