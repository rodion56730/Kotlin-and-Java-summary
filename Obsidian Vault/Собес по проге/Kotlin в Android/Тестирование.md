# Тестирование в Android: Unit и Instrumented тесты

## 1. Пирамида тестирования

```
        /\
       /  \
      / UI \        ← Instrumented (медленно, нужен эмулятор)
     /──────\
    /Integra-\
   / tion     \     ← Частично на JVM, частично Instrumented
  /────────────\
 /  Unit Tests  \   ← JVM, быстро, без устройства
/────────────────\
```

- **Unit тесты** (~70%): тестируют одну функцию/класс в изоляции. Запускаются на JVM.
- **Integration тесты** (~20%): тестируют взаимодействие нескольких классов.
- **UI тесты** (~10%): тестируют весь экран. Нужен эмулятор или устройство.

## 3. Unit тесты: ViewModel

Самое частое на собеседовании — "напиши тест для ViewModel".

```kotlin
class UserViewModelTest {

    // Заменяем Main Dispatcher на тестовый (нет Android в JVM)
    @get:Rule
    val mainDispatcherRule = MainDispatcherRule()

    // Мок репозитория — не делаем реальных запросов в сеть
    private val userRepository: UserRepository = mockk()
    private lateinit var viewModel: UserViewModel

    @Before
    fun setUp() {
        viewModel = UserViewModel(userRepository)
    }

    @Test
    fun `loadUser success updates uiState to Success`() = runTest {
        // ARRANGE (Подготовка)
        val fakeUser = User(id = "1", name = "Алексей")
        coEvery { userRepository.getUser("1") } returns Result.success(fakeUser)

        // ACT (Действие)
        viewModel.loadUser("1")

        // ASSERT (Проверка)
        val state = viewModel.uiState.value
        assertTrue(state is UiState.Success)
        assertEquals("Алексей", (state as UiState.Success).user.name)
    }

    @Test
    fun `loadUser failure updates uiState to Error`() = runTest {
        coEvery { userRepository.getUser(any()) } returns Result.failure(Exception("Нет сети"))

        viewModel.loadUser("1")

        assertTrue(viewModel.uiState.value is UiState.Error)
    }
}
```

## 4. MainDispatcherRule — обязательная вещь

В тестах нет Android-окружения, поэтому `Dispatchers.Main` не работает. Нужно его подменить.

```kotlin
// Создаём правило (можно вынести в отдельный файл и переиспользовать)
class MainDispatcherRule(
    val testDispatcher: TestDispatcher = UnconfinedTestDispatcher()
) : TestWatcher() {
    override fun starting(description: Description) {
        Dispatchers.setMain(testDispatcher)
    }
    override fun finished(description: Description) {
        Dispatchers.resetMain()
    }
}
```

## 5. MockK — мокирование в Kotlin

**MockK** — это библиотека для создания "фейковых" объектов, которые ведут себя так, как ты скажешь.

```kotlin
// Создание мока
val repository: UserRepository = mockk()

// Настройка поведения обычной функции
every { repository.isLoggedIn() } returns true

// Настройка suspend-функции (coEvery для корутин)
coEvery { repository.getUser("1") } returns Result.success(fakeUser)

// Проверка, что метод был вызван
verify { repository.isLoggedIn() }
coVerify { repository.getUser("1") }

// Проверка, что метод НЕ был вызван
verify(exactly = 0) { repository.getUser(any()) }

// Мок, который можно не настраивать (все методы возвращают дефолт)
val repository: UserRepository = mockk(relaxed = true)
```

## 6. Тестирование Flow с Turbine

Flow сложно тестировать через `collect` вручную. Библиотека **Turbine** делает это удобно.

```kotlin
@Test
fun `users flow emits correct items`() = runTest {
    coEvery { repository.getUsersFlow() } returns flowOf(
        listOf(User("1", "Алексей")),
        listOf(User("1", "Алексей"), User("2", "Мария"))
    )

    viewModel.usersFlow.test {
        val firstEmission = awaitItem()
        assertEquals(1, firstEmission.size)

        val secondEmission = awaitItem()
        assertEquals(2, secondEmission.size)

        awaitComplete() // Убеждаемся, что Flow завершился
    }
}
```

## 7. Тестирование Use Case

UseCase тестировать проще всего — это чистый Kotlin без Android-зависимостей.

```kotlin
class GetUserWithPostsUseCaseTest {

    private val userRepository: UserRepository = mockk()
    private val postsRepository: PostsRepository = mockk()
    private val useCase = GetUserWithPostsUseCase(userRepository, postsRepository)

    @Test
    fun `returns combined user and posts on success`() = runTest {
        val fakeUser = User("1", "Алексей")
        val fakePosts = listOf(Post("1", "Первый пост"))

        coEvery { userRepository.getUser("1") } returns Result.success(fakeUser)
        coEvery { postsRepository.getPostsByUser("1") } returns Result.success(fakePosts)

        val result = useCase("1")

        assertTrue(result.isSuccess)
        assertEquals("Алексей", result.getOrNull()?.user?.name)
        assertEquals(1, result.getOrNull()?.posts?.size)
    }
}
```

## 8. UI тесты с Compose Testing

Для Compose есть удобный API тестирования — без Espresso.

```kotlin
@RunWith(AndroidJUnit4::class)
class UserScreenTest {

    @get:Rule
    val composeTestRule = createComposeRule()

    @Test
    fun loadingState_showsProgressIndicator() {
        composeTestRule.setContent {
            UserScreen(uiState = UiState.Loading)
        }

        // Проверяем наличие элемента по тегу
        composeTestRule.onNodeWithTag("loading_indicator").assertIsDisplayed()
    }

    @Test
    fun successState_showsUserName() {
        composeTestRule.setContent {
            UserScreen(uiState = UiState.Success(User("1", "Алексей")))
        }

        // Проверяем наличие текста
        composeTestRule.onNodeWithText("Алексей").assertIsDisplayed()
    }

    @Test
    fun button_click_triggersCallback() {
        var clicked = false
        composeTestRule.setContent {
            Button(onClick = { clicked = true }) { Text("Нажать") }
        }

        composeTestRule.onNodeWithText("Нажать").performClick()
        assertTrue(clicked)
    }
}
```

## 9. Соглашение по именованию тестов

Хорошее имя теста отвечает на три вопроса: **что тестируем**, **при каких условиях**, **что ожидаем**.

```kotlin
// Плохо:
@Test fun test1() { ... }
@Test fun testLoadUser() { ... }

// Хорошо (шаблон: `методТестирования состояние ожидаемыйРезультат`):
@Test fun `loadUser when network error updates state to Error`() { ... }
@Test fun `loadUser success emits user name to UI`() { ... }
@Test fun `getUser returns cached data when offline`() { ... }
```

## Ключевые фразы для собеседования

- **"Unit тест — тестирую один класс в изоляции, зависимости заменяю моками"**.
- **"MockK для suspend-функций: `coEvery` вместо `every`"** — частая ошибка новичков.
- **"MainDispatcherRule нужен, чтобы `Dispatchers.Main` работал в JVM-тестах"**.
- **"Turbine упрощает тестирование Flow — `awaitItem()` вместо ручного collect"**.
- **"Пирамида: 70% Unit, 20% Integration, 10% UI"** — почему не всё тестируем через Espresso.
- **"AAA паттерн: Arrange, Act, Assert"** — структура каждого теста.
