# Kotlin vs Java: Основные различия и система типов

В Kotlin, в отличие от Java, отсутствует явное разделение на примитивные типы (`int`, `double`) и классы-обертки (`Integer`, `Double`). На уровне исходного кода **все является объектом**. Однако для оптимизации производительности компилятор Kotlin автоматически использует примитивы JVM там, где это возможно.

## Единая система типов (No Primitive Types)

В Kotlin вы всегда используете такие типы, как `Int`, `Float`, `Boolean`. Вам не нужно выбирать между `int` и `Integer`.

- **Как это работает:**
    
    - Компилятор анализирует код: если переменная не может быть `null` и не используется в коллекциях, она будет скомпилирована в обычный **примитив Java** для скорости.
        
    - Если переменная помечена как **Nullable** (`Int?`), она автоматически станет объектом-оберткой (аналог `Integer`).
        

---

## Сравнение механизмов

|**Характеристика**|**Java**|**Kotlin**|
|---|---|---|
|**Примитивы**|Есть (`int`, `char`, `boolean`)|Нет (используются `Int`, `Char`, `Boolean`)|
|**Автоупаковка**|Автоматическая (между `int` и `Integer`)|Скрыта (компилятор сам решает, когда нужен объект)|
|**Null Safety**|Любой объект может быть `null`|Разделение на `Type` (не-null) и `Type?` (nullable)|
|**Приведение типов**|Явное: `(long) myInt`|Через методы: `myInt.toLong()`|
|**Исключения**|Есть Checked и Unchecked|Только Unchecked (нет обязательных `try-catch`)|

## Null Safety (Безопасность null)

Это фундаментальное различие. В Java `Integer` может хранить `null`, что часто ведет к `NullPointerException` (NPE) при анбоксинге. В Kotlin эта проблема решается на уровне компиляции.

- **Non-nullable (По умолчанию):**
```kotlin
var a: Int = 10 
// a = null // Ошибка компиляции!
```

- **Nullable (Аналог объекта-обертки Java):**

```kotlin
var b: Int? = 10 
b = null // Разрешено. В байт-коде это превратится в Integer.
```

## Ключевые архитектурные отличия

### 1. Smart Casts (Умные приведения)

В Java после проверки `instanceof` нужно явно приводить тип. В Kotlin после проверки `is` объект автоматически считается нужным типом в области видимости.

```kotlin
if (obj is String) {
    println(obj.length) // obj уже String, каст (String)obj не нужен
}
```

### 2. Свойства (Properties) vs Поля (Fields)

В Java мы пишем поля, геттеры и сеттеры. В Kotlin мы объявляем свойства, а доступ к ним выглядит как обращение к полю, но под капотом вызываются методы.

Kotlin

```kotlin
class User {
    var name: String = "" // Автоматически создаются геттер и сеттер
}
```

### 3. Отсутствие Static

В Kotlin нет ключевого слова `static`. Его функции выполняют:

- **Top-level functions**: Функции вне класса (в файле).
    
- **Companion Objects**: Объекты внутри класса для доступа к "статическим" членам.
    

### 4. Краткость кода (Data Classes)

Там, где в Java требуется 50 строк кода (геттеры, сеттеры, `equals`, `hashCode`), в Kotlin достаточно одной:

Kotlin

```kotlin
data class User(val id: Int, val name: String)
```

## Ключевые моменты

- **Совместимость:** Kotlin полностью совместим с Java. Вы можете вызывать Java-код из Kotlin и наоборот.
    
- **Безопасность:** Система типов Kotlin спроектирована так, чтобы минимизировать ошибки времени выполнения (особенно NPE).
    
- **Лаконичность:** Отсутствие "проверяемых" исключений (Checked Exceptions) и наличие свойств уменьшает количество шаблонного (boilerplate) кода.
