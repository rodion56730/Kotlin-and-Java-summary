В Kotlin система классов гораздо гибче, чем в Java. Вместо того чтобы писать всё вручную, вы выбираете конкретный «инструмент» под задачу.

---

# Типы классов в Kotli

В Kotlin классы по умолчанию являются **final** (их нельзя наследовать) и **public**. Это сделано для безопасности архитектуры.

## 1. Data Classes (Классы данных)

Это "звезда" Kotlin. Предназначены исключительно для хранения данных. Компилятор автоматически генерирует: `equals()`, `hashCode()`, `toString()`, `copy()` и `componentN()` (для деструктуризации).

- **Особенности:**
    
    - Должен иметь хотя бы один параметр в конструкторе.
        
    - Все параметры конструктора должны быть помечены `val` или `var`.
        
- **Пример:**


```Kotlin
data class User(val id: Int, val name: String)

val user1 = User(1, "Alex")
val user2 = user1.copy(name = "Max") // Клонируем с изменением одного поля
println(user1) // Выведет: User(id=1, name=Alex) — красиво и понятно
```

## 2. Sealed Classes (Запечатанные классы)

Это "Enum на стероидах". Они ограничивают иерархию наследования: все подклассы должны быть известны на этапе компиляции. Идеальны для обработки состояний (например, сетевого запроса).

- **Пример:**


```Kotlin
sealed class Result
class Success(val data: String) : Result()
class Error(val message: String) : Result()

// В сочетании с 'when' компилятор проверит, все ли варианты обработаны
fun handle(result: Result) = when(result) {
    is Success -> println("Ура: ${result.data}")
    is Error -> println("Ошибка: ${result.message}")
    // 'else' не нужен, так как иерархия закрыта
}
```

## 3. Enum Classes (Перечисления)

Почти такие же, как в Java, но с более приятным синтаксисом. Могут иметь свойства и методы.

```Kotlin
enum class Priority(val value: Int) {
    LOW(1), MEDIUM(5), HIGH(10);
    
    fun isUrgent() = this == HIGH
}
```

## 4. Object (Синглетоны)

В Kotlin нет нужды писать сложный код для паттерна Singleton. Ключевое слово `object` создает класс и его единственный экземпляр одновременно.

Kotlin

```Kotlin
object DatabaseConfig {
    val url = "jdbc:mysql://localhost:3306"
    fun connect() = println("Connecting to $url")
}
// Вызов: DatabaseConfig.connect() (создавать экземпляр через new не надо)
```

## 5. Companion Object (Объект-спутник)

Заменяет `static` из Java. Это объект, привязанный к конкретному классу.

```Kotlin
class MyServer {
    companion object {
        const val DEFAULT_PORT = 8080
        fun create() = MyServer()
    }
}
// Вызов: MyServer.DEFAULT_PORT
```

## 6. Inner и Nested классы

- **Nested (Вложенный)**: По умолчанию класс внутри класса не имеет доступа к членам внешнего класса (аналог `static class` в Java).
    
- **Inner (Внутренний)**: Чтобы получить доступ к внешнему классу, нужно добавить слово `inner`.
    

---

### Сводная таблица для конспекта

|**Тип класса**|**Главная фишка**|**Когда использовать**|
|---|---|---|
|**Data**|Авто-генерация методов|Для моделей данных, DTO, сущностей|
|**Sealed**|Ограниченное наследование|Для сложных состояний (UI States, Errors)|
|**Enum**|Список констант|Для простых перечислений (Дни недели, Цвета)|
|**Object**|Singleton из коробки|Для конфигов, утилит, логгеров|
|**Companion**|Замена static|Для фабричных методов и констант класса|

### Пример кода для Obsidian

```Kotlin
// Класс данных
data class Product(val name: String, val price: Double)

// Запечатанный класс для состояний экрана
sealed class ScreenState {
    object Loading : ScreenState()
```

# Интерфейсы в Kotlin

Интерфейсы в Kotlin очень похожи на интерфейсы Java 8+, но они более гибкие: они могут содержать как объявления абстрактных методов, так и их реализацию (методы по умолчанию).

## 1. Базовый синтаксис

Интерфейс объявляется ключевым словом `interface`. Класс реализует интерфейс через двоеточие `:` (так же, как и при наследовании классов).


```Kotlin
interface MyInterface {
    fun bar()           // Абстрактный метод (нужно реализовать)
    fun foo() {         // Метод с реализацией по умолчанию
        println("foo")
    }
}

class Child : MyInterface {
    override fun bar() {
        println("bar implementation")
    }
}
```

## 2. Свойства в интерфейсах (Properties)

В отличие от Java, в интерфейсах Kotlin можно объявлять свойства. Однако они **не могут хранить состояние** (у них нет backing field). Это значит, что свойство в интерфейсе либо должно быть абстрактным, либо иметь кастомный геттер.


```Kotlin
interface User {
    val nickname: String // Абстрактное свойство

    val email: String
        get() = "$nickname@example.com" // Свойство с геттером
}

class FacebookUser(override val nickname: String) : User
```

## 3. Решение конфликтов (Multiple Inheritance)

Если класс реализует два интерфейса с одинаковыми методами (у которых есть реализация), Kotlin **заставляет** вас переопределить этот метод, чтобы избежать неопределенности.

```Kotlin
interface A {
    fun doSomething() { println("A") }
}

interface B {
    fun doSomething() { println("B") }
}

class MyClass : A, B {
    override fun doSomething() {
        super<A>.doSomething() // Явный вызов метода из A
        super<B>.doSomething() // Явный вызов метода из B
    }
}
```

## 4. Делегирование интерфейсов (Interface Delegation)

Это уникальная фишка Kotlin. Она позволяет передать реализацию интерфейса другому объекту с помощью ключевого слова `by`. Это заменяет паттерн «Композиция вместо наследования».


```Kotlin
interface SoundBehavior {
    fun makeSound()
}

class Bark : SoundBehavior {
    override fun makeSound() = println("Bark!")
}

// Класс Dog реализует интерфейс SoundBehavior, 
// делегируя всю работу объекту bark
class Dog(bark: Bark) : SoundBehavior by bark

fun main() {
    val dog = Dog(Bark())
    dog.makeSound() // Напечатает "Bark!"
}
```

---

### Сравнение с Java

|**Функция**|**Java**|**Kotlin**|
|---|---|---|
|**Реализация по умолчанию**|Ключевое слово `default`|Не требуется (просто пишем тело)|
|**Свойства**|Только константы (`static final`)|Можно объявлять свойства с геттерами|
|**Делегирование**|Нужно писать вручную|Ключевое слово `by`|
|**Конфликты имен**|Решаются через `Interface.super.method()`|Решаются через `super<Interface>.method()`|

---

### Ключевые моменты

- **Stateless**: Интерфейсы не могут хранить данные (поля), только логику доступа к ним.
    
- **Гибкость**: Использование `by` позволяет создавать мощные конструкции без раздувания иерархии классов.
    
- **Override**: Как и с классами, слово `override` обязательно для реализации функций интерфейса.

# Абстрактные классы в Kotlin

Абстрактный класс — это класс, экземпляр которого нельзя создать. Он служит "шаблоном" или базой для других классов. Объявляется ключевым словом `abstract`.

## 1. Основные правила

- **Не нужно `open`**: Абстрактные классы и их абстрактные методы открыты для наследования по умолчанию. Слово `open` писать не нужно.
    
- **Смешивание логики**: В абстрактном классе могут быть как методы с реализацией, так и абстрактные методы (без тела).
    
- **Хранение состояния**: В отличие от интерфейсов, абстрактные классы **могут иметь поля (состояние)**.
    

## 2. Пример кода


```Kotlin
abstract class Vehicle(val brand: String) {
    
    // Абстрактный метод (должен быть переопределен в потомках)
    abstract fun drive()

    // Обычный метод (уже имеет логику, можно не переопределять)
    fun stop() {
        println("$brand останавливается")
    }
}

class Car(brand: String) : Vehicle(brand) {
    override fun drive() {
        println("$brand едет по дороге")
    }
}
```

---

## 3. Абстрактные свойства

Как и методы, свойства в классе могут быть абстрактными. Это обязывает наследников инициализировать их.

```Kotlin
abstract class BaseScreen {
    abstract val title: String // Наследник обязан дать название
}

class ProfileScreen : BaseScreen() {
    override val title = "Профиль пользователя"
}
```

---

## 4. Сравнение: Абстрактный класс vs Интерфейс

Это классический вопрос на собеседованиях.

|**Характеристика**|**Абстрактный класс**|**Интерфейс**|
|---|---|---|
|**Состояние (Fields)**|Может хранить данные в полях|Не может хранить данные (только геттеры)|
|**Наследование**|Можно наследоваться только от одного|Можно реализовать много интерфейсов|
|**Конструктор**|Может иметь конструктор с параметрами|Не имеет конструкторов|
|**Цель**|Общая идентичность ("Является кем-то")|Общее поведение ("Умеет что-то")|

---

### Когда что выбирать?

1. **Интерфейс**: Если вам нужно описать _способность_ объекта (например, `Flyable`, `Clickable`).
    
2. **Абстрактный класс**: Если у группы классов есть общая _сущность_ и общие данные (например, у всех `Shape`есть `color`, а у всех `Animal` есть `age`).
    

---

### Ключевые моменты

- Используйте `abstract`, если класс не имеет смысла без реализации в наследниках.
    
- Абстрактные члены класса не имеют тела.
    
- Если метод в абстрактном классе не помечен как `abstract`, он считается `final` (если не добавить `open`).
