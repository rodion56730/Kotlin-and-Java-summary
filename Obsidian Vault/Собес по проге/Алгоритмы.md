# 🧠 Конспект: Алгоритмы и структуры данных в Kotlin

## 📊 Сложность алгоритмов (Big O Cheat Sheet)

| **Алгоритм**      | **Лучшее время** | **Среднее / Худшее** | **Память**  | **Особенности**                         |
| ----------------- | ---------------- | -------------------- | ----------- | --------------------------------------- |
| **Bubble Sort**   | $O(n)$           | $O(n^2)$             | $O(1)$      | Самый медленный, только для обучения    |
| **Quick Sort**    | $O(n \log n)$    | $O(n^2)$             | $O(\log n)$ | Эффективен, "Разделяй и властвуй"       |
| **Merge Sort**    | $O(n \log n)$    | $O(n \log n)$        | $O(n)$      | Стабилен, требует доп. память           |
| **Binary Search** | $O(1)$           | $O(\log n)$          | $O(1)$      | **Только для отсортированных** массивов |
| **Linear Search** | $O(1)$           | $O(n)$               | $O(1)$      | Простой перебор                         |

## 1. Сортировка (Sorting)

### Быстрая сортировка (Quick Sort)

Самая популярная. Выбираем `pivot` (опорный элемент) и распределяем элементы: меньше него — влево, больше — вправо. Рекурсивно повторяем.
```Kotlin
fun quickSort(list: List<Int>): List<Int> {
    if (list.size <= 1) return list
    val pivot = list[list.size / 2]
    val less = list.filter { it < pivot }
    val equal = list.filter { it == pivot }
    val greater = list.filter { it > pivot }
    return quickSort(less) + equal + quickSort(greater)
}
```

### Сортировка слиянием (Merge Sort)

Делит массив пополам до единичных элементов, а затем сливает их в правильном порядке.

```Kotlin
fun mergeSort(list: List<Int>): List<Int> {
    if (list.size <= 1) return list
    val mid = list.size / 2
    val left = mergeSort(list.subList(0, mid))
    val right = mergeSort(list.subList(mid, list.size))
    return merge(left, right)
}

private fun merge(left: List<Int>, right: List<Int>): List<Int> {
    var i = 0; var j = 0
    val result = mutableListOf<Int>()
    while (i < left.size && j < right.size) {
        if (left[i] <= right[j]) result.add(left[i++]) else result.add(right[j++])
    }
    return result + left.drop(i) + right.drop(j)
}
```

## 2. Поиск (Searching)

### Бинарный поиск (Binary Search)

Делим отсортированный массив пополам. Если искомое больше среднего — ищем в правой половине, меньше — в левой.

```Kotlin
fun binarySearch(arr: IntArray, target: Int): Int {
    var low = 0
    var high = arr.size - 1
    while (low <= high) {
        val mid = (low + high) / 2
        when {
            arr[mid] == target -> return mid
            arr[mid] < target -> low = mid + 1
            else -> high = mid - 1
        }
    }
    return -1
}
```

## 3. Обход графов и деревьев

### BFS (Поиск в ширину)

Использует **Очередь (Queue)**. Ищет кратчайший путь в ненагруженном графе (по уровням).
```Kotlin
fun bfs(startNode: Node) {
    val queue: Queue<Node> = LinkedList<Node>().apply { add(startNode) }
    val visited = mutableSetOf<Node>(startNode)

    while (queue.isNotEmpty()) {
        val node = queue.poll()
        node.neighbors.forEach { neighbor ->
            if (neighbor !in visited) {
                visited.add(neighbor)
                queue.add(neighbor)
            }
        }
    }
}
```

### DFS (Поиск в глубину)

Использует **Рекурсию** или Стек. Идет вглубь до тупика, потом возвращается.

```Kotlin
fun dfs(node: Node, visited: MutableSet<Node> = mutableSetOf()) {
    if (node in visited) return
    visited.add(node)
    node.neighbors.forEach { dfs(it, visited) }
}
```

## 4. Оптимизация и техники

### Динамическое программирование (Фибоначчи с Мемоизацией)

Вместо $O(2^n)$ получаем $O(n)$, сохраняя результаты подзадач.

```Kotlin
val memo = mutableMapOf<Int, Long>()
fun fib(n: Int): Long {
    if (n <= 1) return n.toLong()
    return memo.getOrPut(n) { fib(n - 1) + fib(n - 2) }
}
```

### Два указателя (Two Pointers)

Эффективный способ обработки массивов за $O(n)$ вместо $O(n^2)$.

```Kotlin
// Найти два числа, дающих в сумме target в отсортированном массиве
fun hasTwoSum(arr: IntArray, target: Int): Boolean {
    var left = 0
    var right = arr.size - 1
    while (left < right) {
        val sum = arr[left] + arr[right]
        if (sum == target) return true
        if (sum < target) left++ else right--
    }
    return false
}
```

### 💡 Шпаргалка: когда что выбирать?

1. **Массив не отсортирован?** Сначала `sort()` (Quick/Merge), потом `binarySearch()`.
    
2. **Нужен кратчайший путь?** `BFS`.
    
3. **Нужно обойти все варианты (комбинации)?** `DFS` / `Backtracking`.
    
4. **Слишком медленно ($O(n^2)$)?** Попробуй `HashMap` или `Two Pointers`.

Алгоритм Дейкстры — это «король» алгоритмов на графах, когда речь идет о поиске **кратчайшего пути в нагруженном графе** (где у каждого ребра/дороги есть свой «вес» — расстояние, цена или время).

Его обязательно спрашивают на собеседованиях уровня Middle, так как он сложнее обычного BFS.

# Алгоритм Дейкстры (Dijkstra's Algorithm)

## 1. Суть алгоритма

В отличие от BFS, который просто считает количество шагов, Дейкстра учитывает **стоимость** каждого шага.

**Принцип работы:**

1. Создаем таблицу расстояний, где до всех узлов расстояние — «бесконечность», а до стартового — 0.
    
2. Выбираем самый «дешевый» узел из еще не посещенных.
    
3. Обновляем расстояния до всех его соседей: если путь через текущий узел короче, чем тот, что записан в таблице — перезаписываем его (это называется **релаксация ребер**).
    
4. Помечаем узел как посещенный и больше к нему не возвращаемся.
    
5. Повторяем, пока не посетим все доступные узлы.
    

> **Важное ограничение:** Алгоритм работает только в графах **без отрицательных весов**.

---

## 2. Реализация на Kotlin

Для эффективности мы будем использовать `PriorityQueue` (приоритетную очередь). Она всегда держит самый «дешевый» узел сверху, что позволяет нам не перебирать все узлы вручную.



```Kotlin
import java.util.PriorityQueue

// Описываем ребро графа: куда ведет и сколько стоит
data class Edge(val to: String, val weight: Int)

fun dijkstra(graph: Map<String, List<Edge>>, start: String): Map<String, Int> {
    // Таблица кратчайших расстояний
    val distances = mutableMapOf<String, Int>().withDefault { Int.MAX_VALUE }
    distances[start] = 0

    // Очередь с приоритетом: сначала извлекаем узел с минимальным весом
    val pq = PriorityQueue<Pair<String, Int>>(compareBy { it.second })
    pq.add(start to 0)

    val visited = mutableSetOf<String>()

    while (pq.isNotEmpty()) {
        val (currentNode, currentDist) = pq.poll()

        if (currentNode in visited) continue
        visited.add(currentNode)

        // Смотрим всех соседей текущего узла
        graph[currentNode]?.forEach { edge ->
            val newDist = currentDist + edge.weight
            
            // Если нашли путь короче — обновляем таблицу и добавляем в очередь
            if (newDist < distances.getValue(edge.to)) {
                distances[edge.to] = newDist
                pq.add(edge.to to newDist)
            }
        }
    }
    return distances
}

fun main() {
    // Пример графа (карта дорог)
    val graph = mapOf(
        "A" to listOf(Edge("B", 4), Edge("C", 2)),
        "B" to listOf(Edge("C", 5), Edge("D", 10)),
        "C" to listOf(Edge("D", 3)),
        "D" to emptyList()
    )

    val results = dijkstra(graph, "A")
    println(results) // Выведет минимальные пути: {A=0, B=4, C=2, D=5}
}
```

---

## 3. Разбор сложности (Big O)

- **Сложность по времени:** $O((V + E) \log V)$, где $V$ — количество вершин, а $E$ — ребер.
    
    - $\log V$ берется за счет использования приоритетной очереди (бинарной кучи).
        
- **Сложность по памяти:** $O(V + E)$ для хранения графа и таблицы расстояний.

## 4. Зачем это Android-разработчику?

1. **Навигация:** Логика поиска пути внутри зданий, торговых центров или на картах.
    
2. **Оптимизация сетей:** Поиск «дешевого» пути передачи данных (например, через какой сервер пробросить запрос).
    
3. **Игры:** Поиск пути (Pathfinding) для мобов или персонажей.
    

### Сравнение с другими поисками:

- **BFS:** Ищет кратчайший путь, только если все ребра равны 1 (без весов).
    
- **Дейкстра:** Ищет кратчайший путь, если веса разные (положительные).
    
- *_A_ (A-star):** «Умный» Дейкстра, который использует эвристику (предположение о расстоянии до цели), чтобы искать еще быстрее.
    

---

### Краткий итог:

- **Дейкстра** — это «жадный» алгоритм. На каждом шаге он берет самый выгодный вариант здесь и сейчас.
    
- Ключ к быстрой работе — **PriorityQueue**.
    
- Не работает с отрицательными весами (для этого есть алгоритм Беллмана-Форда).
