## 🗣 ЧАСТЬ 1 — РАССКАЗ О ПРОЕКТЕ (5-7 минут)

### ❓ Расскажи о своём тестовом задании

**Готовый рассказ (3-4 мин):**

Я разработал Android-приложение для просмотра контактов устройства с возможностью удаления дублей. Приложение состоит из двух частей: основного приложения и отдельного сервиса, связанных через AIDL.

**Архитектура:** MVVM + Clean Architecture. UI-слой — Fragment + RecyclerView, ViewModel хранит StateFlow со списком контактов. Repository читает контакты через ContentResolver и ContactsContract. Операции удаления дублей вынесены в отдельный Bound Service, общение с которым идёт через AIDL-интерфейс.

**Логика поиска дублей:** Контакты считаются дублями если у них полностью совпадают все поля — имя, телефон, email. Группирую контакты по ключу из конкатенации полей, нахожу группы размером > 1 и удаляю все кроме первого через ContentResolver с разрешением WRITE_CONTACTS.

**Уведомление UI:** Сервис через AIDL возвращает один из трёх статусов — успех, ошибка, дубли не найдены. UI показывает соответствующий Toast и обновляет список.

### ❓ Почему именно AIDL, а не Messenger / BroadcastReceiver?

**Ответ:** AIDL выбран по условию задания — все операции должны быть в отдельном сервисе с IPC через AIDL. По техническим причинам:

- AIDL поддерживает типизированный интерфейс — компилятор генерирует Stub и Proxy
- Messenger — обёртка над AIDL, но для однопоточной обработки. AIDL работает в thread pool Binder'а — подходит для параллельных вызовов
- BroadcastReceiver — не для долгих операций и слабее в плане типизации

### ❓ Что такое AIDL? Как работает под капотом?

**Ответ:** AIDL — Android Interface Definition Language. Язык описания интерфейса для IPC (межпроцессного взаимодействия) через Binder.

**Как работает:**

1. Пишем `.aidl` файл с интерфейсом
2. Компилятор генерирует Java-класс с `Stub` (серверная сторона) и `Proxy` (клиентская)
3. Сервис возвращает `Stub.asBinder()` в `onBind()`
4. Клиент получает `Proxy` через `ServiceConnection.onServiceConnected()`
5. Вызов метода на `Proxy` → сериализация аргументов в `Parcel` → передача через Binder → десериализация и вызов в `Stub` на стороне сервиса

**На стороне сервиса** методы выполняются в Binder thread pool — не в Main Thread!

---

### ❓ Что такое `oneway` в AIDL?

**Ответ:** `oneway` делает вызов асинхронным — клиент не блокируется и не ждёт результата:

```aidl
interface IContactService {
    oneway void deleteDuplicates(); // fire-and-forget
    void getStatus(); // синхронный — клиент ждёт
}
```

Ограничение: `oneway` методы не могут возвращать значения и не могут иметь `out` параметры. Используется когда клиенту не нужен результат немедленно.

---

### ❓ Жизненный цикл Bound Service

**Ответ:**

```
onCreate() → onBind() → [клиент подключён, работает]
→ onUnbind() → onDestroy()
```

- `onBind()` — возвращает IBinder клиенту. Вызывается при первом подключении
- `onUnbind()` — все клиенты отвязались. Если вернуть `true` — при повторном bind вызовется `onRebind()`
- Сервис живёт пока есть хотя бы один привязанный клиент
- Если клиент умирает не вызвав `unbindService()` — система сама отвязывает

### ❓ Чем BoundService отличается от StartedService?

**Ответ:**

||Started Service|Bound Service|
|---|---|---|
|Запуск|`startService()`|`bindService()`|
|Жизнь|Пока не вызовут `stopSelf()` / `stopService()`|Пока есть клиенты|
|Результат|Нет прямого канала|Через IBinder / AIDL|
|Пример|Загрузка файла|AIDL-сервис контактов|

Можно комбинировать: `startService()` + `bindService()` — сервис живёт даже без клиентов.

### ❓ Что такое ContentProvider? Как удалить контакт?

**Ответ:** ContentProvider — компонент Android для предоставления данных другим приложениям через URI. Контакты системы доступны через `ContactsContract`.

Удаление контакта:

```kotlin
val uri = ContentUris.withAppendedId(
```

Нужно разрешение `WRITE_CONTACTS`. Удаление через `Contacts.CONTENT_URI` каскадно удаляет все данные контакта (телефоны, email и т.д.).

### ❓ Что такое Cursor и как правильно с ним работать?

**Ответ:** `Cursor` — указатель на набор строк из запроса к ContentProvider или БД. Держит соединение с ресурсом — его **обязательно закрывать**.

```kotlin
// Правильно — use() закрывает автоматически (Closeable)
contentResolver.query(uri, ...)?.use { cursor ->
    while (cursor.moveToNext()) {
        val name = cursor.getString(
            cursor.getColumnIndexOrThrow(ContactsContract.Contacts.DISPLAY_NAME)
        )
    }
}
// Неправильно — утечка если забыть close() или выбросится исключение
val cursor = contentResolver.query(...)
// ... cursor.close() — может не вызваться
```

### ❓ Что такое Parcelable и зачем нужен для AIDL?

**Ответ:** `Parcelable` — Android-интерфейс для сериализации объектов в `Parcel` (бинарный буфер) для передачи через IPC.

Почему не `Serializable`:

- Serializable использует reflection — медленно, много мусора
- Parcelable — ручная сериализация, в 10x быстрее
- AIDL требует Parcelable для кастомных объектов

```kotlin
@Parcelize // kotlin-parcelize plugin — генерирует код автоматически
data class Contact(val id: Long, val name: String, val phone: String) : Parcelable
```

В AIDL файле объявляем: `parcelable Contact;`

### ❓ Что такое suspend-функция под капотом?

**Ответ:** `suspend` — маркер для компилятора. Компилятор превращает функцию в state machine:

```kotlin
suspend fun loadUser(): User { ... }
// Компилятор генерирует примерно:
fun loadUser(continuation: Continuation<User>): Any
```

`Continuation` хранит состояние функции (как save-point в игре) — текущую позицию, локальные переменные. Когда функция "приостанавливается" (например на `delay`) — она освобождает поток и сохраняет Continuation. Когда результат готов — выполнение продолжается с того же места.

**Главное:** `suspend` не создаёт новый поток. Поток освобождается и может делать другую работу.

### ❓ launch vs async — в чём разница?

**Ответ:**

||`launch`|`async`|
|---|---|---|
|Возвращает|`Job`|`Deferred<T>`|
|Результат|Нет|Есть — через `await()`|
|Использование|fire-and-forget|нужен результат|

```kotlin
// launch — просто запустить
viewModelScope.launch {
    repository.deleteContacts()
}

// async — параллельные запросы с результатом
viewModelScope.launch {
    val contactsDeferred = async { repository.getContacts() }
    val settingsDeferred = async { repository.getSettings() }
    // Оба запроса идут параллельно — 1 сек вместо 2
    val contacts = contactsDeferred.await()
    val settings = settingsDeferred.await()
}
```

### ❓ StateFlow vs SharedFlow — когда что использовать?

**Ответ:**

||`StateFlow`|`SharedFlow`|
|---|---|---|
|Хранит последнее значение|Да (replay=1)|Нет по умолчанию|
|Начальное значение|Обязательно|Нет|
|Использование|Состояние UI|Одноразовые события|
|Пример|loading/success/error|Toast, Navigation|

```kotlin
// StateFlow — что сейчас показывать
private val _contacts = MutableStateFlow<List<Contact>>(emptyList())
val contacts = _contacts.asStateFlow()

// SharedFlow — что произошло (не хотим повтора после поворота)
private val _events = MutableSharedFlow<UiEvent>()
val events = _events.asSharedFlow()

// Emit события
_events.emit(UiEvent.ShowToast("Дубли удалены"))
```

После поворота экрана StateFlow отдаст последнее состояние (правильно), SharedFlow — нет (тост не покажется повторно — тоже правильно).

### ❓ Что такое viewModelScope и откуда берётся?

**Ответ:** `viewModelScope` — extension property из `androidx.lifecycle:lifecycle-viewmodel-ktx`. Это `CoroutineScope` с `SupervisorJob + Dispatchers.Main`.

```kotlin
// Под капотом примерно так:
val ViewModel.viewModelScope: CoroutineScope
    get() = CloseableCoroutineScope(SupervisorJob() + Dispatchers.Main.immediate)

// В onCleared() ViewModel автоматически вызывает:
viewModelScope.cancel()
```

**Почему SupervisorJob:** если одна корутина упала — остальные продолжают работать. Одна неудачная сетевая операция не убивает весь ViewModel.

**Почему не GlobalScope:** GlobalScope не привязан к lifecycle — корутина переживёт уничтожение экрана и будет держать ссылки → утечка памяти.

### ❓ Как обрабатывать ошибки в корутинах?

**Ответ:** Три подхода:

```kotlin
// 1. try-catch — самый простой и понятный
viewModelScope.launch {
    try {
        val result = repository.loadData()
        _state.value = UiState.Success(result)
    } catch (e: IOException) {
        _state.value = UiState.Error(e.message)
    }
}

// 2. CoroutineExceptionHandler — для launch (не для async)
val handler = CoroutineExceptionHandler { _, throwable ->
    _state.value = UiState.Error(throwable.message)
}
viewModelScope.launch(handler) {
    repository.loadData()
}

// 3. runCatching — возвращает Result<T>
viewModelScope.launch {
    val result = runCatching { repository.loadData() }
    result.onSuccess { _state.value = UiState.Success(it) }
    result.onFailure { _state.value = UiState.Error(it.message) }
}
```

**Важно:** `CoroutineExceptionHandler` не работает с `async` — там исключение держится в `Deferred` до вызова `await()`, нужен `try-catch` вокруг `await()`.

### ❓ data class vs sealed class — когда что?

**Ответ:**

**data class** — когда нужен объект-данные с equals/hashCode/copy/toString:

```kotlin
data class Contact(val id: Long, val name: String, val phone: String)
```

**sealed class** — закрытая иерархия, все подтипы известны компилятору. Идеально для UiState и Result:

```kotlin
sealed class UiState {
    object Loading : UiState()
    data class Success(val contacts: List<Contact>) : UiState()
    data class Error(val message: String) : UiState()
}

// when обязан покрыть все варианты — компилятор проверяет
when (state) {
    is UiState.Loading -> showLoading()
    is UiState.Success -> showContacts(state.contacts)
    is UiState.Error -> showError(state.message)
}
```

### ❓ Чем object отличается от companion object?

**Ответ:**

`object` — синглтон, создаётся при первом обращении:

```kotlin
object ContactsHelper {
    fun normalize(phone: String) = phone.replace(" ", "")
}
ContactsHelper.normalize("+7 999 123")
```

`companion object` — "статические" члены класса. Привязан к конкретному классу:

```kotlin
class Contact(val id: Long) {
    companion object {
        const val UNKNOWN_ID = -1L
        fun empty() = Contact(UNKNOWN_ID)
    }
}
Contact.empty()
Contact.UNKNOWN_ID
```

В Java `companion object` доступен через `Contact.Companion.empty()`, если нет `@JvmStatic`.

### ❓ Что такое extension functions и как они компилируются?

**Ответ:** Extension function добавляет метод к существующему классу без наследования:

```kotlin
fun String.isValidPhone(): Boolean {
    return matches(Regex("\\+?[0-9]{10,13}"))
}
"+79991234567".isValidPhone() // true
```

Компилируется в статическую функцию — никакой магии:

```java
// Под капотом:
public static boolean isValidPhone(String $this) {
    return $this.matches("\\+?[0-9]{10,13}");
}
```

**Ограничения:** не имеет доступа к `private` и `protected` членам. Если в классе уже есть метод с той же сигнатурой — победит метод класса.

### ❓ Null Safety — как работает под капотом?

**Ответ:** Kotlin добавляет `@Nullable` / `@NotNull` аннотации в байткод. Компилятор вставляет null-проверки:

```kotlin
fun greet(name: String) { // @NotNull
    println("Hello, $name")
}
// В байткоде: if (name == null) throw NullPointerException(...)
```

Операторы:

- `?.` — safe call: `user?.name` = `if (user != null) user.name else null`
- `?:` — Elvis: `user?.name ?: "Unknown"` = `if (user?.name != null) user.name else "Unknown"`
- `!!` — assertion: выбрасывает `NullPointerException` если null. Использовать только когда 100% уверен
- `let` — `user?.let { send(it) }` — выполняет блок только если не null

### ❓ Чем Thread отличается от Runnable?

**Ответ:** `Runnable` — функциональный интерфейс с методом `run()`. Описывает задачу. `Thread` — класс который запускает задачу в новом потоке.

```java
// Runnable — только задача
Runnable task = () -> System.out.println("Working");

// Thread — выполнение
Thread thread = new Thread(task);
thread.start();

// Почему лучше Runnable:
// 1. Можно передать в ThreadPool
// 2. Класс не привязан к Thread — можно наследоваться от другого класса
// 3. Переиспользование задачи в разных потоках
```

---

### ❓ Что такое volatile и когда нужен?

**Ответ:** `volatile` гарантирует что значение переменной всегда читается из основной памяти, а не из кэша CPU потока:

```java
// Без volatile — второй поток может читать устаревшее значение из кэша
volatile boolean isRunning = true;

// Поток 1:
isRunning = false;

// Поток 2 — без volatile может не увидеть изменение:
while (isRunning) { ... }
```

**Ограничение:** `volatile` не атомарен для составных операций. `i++` — это три операции (read, increment, write). Для этого нужен `AtomicInteger` или `synchronized`.

### ❓ HashMap — почему не потокобезопасен?

**Ответ:** `HashMap` не синхронизирован. При параллельном resize (увеличении capacity) возможен бесконечный цикл (в Java 7) или `ConcurrentModificationException` (Java 8+).

Решения:

- `Collections.synchronizedMap(map)` — оборачивает в synchronized, но блокирует целиком
- `ConcurrentHashMap` — сегментная блокировка (Java 7) / CAS + synchronized на bucket (Java 8). Быстрее для конкурентного чтения
- В Kotlin в корутинах — `Mutex` или переработать логику чтобы не нужен был shared state

---

### ❓ equals() и hashCode() — контракт

**Ответ:** **Контракт:**

1. Если `a.equals(b)` → то `a.hashCode() == b.hashCode()` (обязательно)
2. Обратное необязательно — разные объекты могут иметь одинаковый hashCode (коллизия)

**Почему важно:**

```java
// HashMap: сначала находит bucket по hashCode, потом equals внутри bucket
map.put(key, value);   // hashCode → bucket → equals → вставка
map.get(key);          // hashCode → bucket → equals → поиск
```

Если переопределить только `equals` без `hashCode` — объект не найдётся в HashMap.

В Kotlin `data class` генерирует оба автоматически по всем полям в конструкторе.

### ❓ Что такое WeakReference и где применяется в Android?

**Ответ:** `WeakReference` — слабая ссылка. GC может собрать объект даже если на него есть WeakReference.

```java
WeakReference<Activity> activityRef = new WeakReference<>(activity);
// Позже:
Activity activity = activityRef.get(); // может быть null если GC собрал
if (activity != null && !activity.isFinishing()) {
    activity.updateUI(result);
}
```

В Android использовался для передачи Activity в AsyncTask — чтобы не держать сильную ссылку и не мешать GC. Сейчас актуально в Handler (если используешь) и в кастомных решениях где нужно избежать утечки контекста.

### ❓ Checked vs Unchecked exceptions

**Ответ:**

||Checked|Unchecked|
|---|---|---|
|Наследует|`Exception`|`RuntimeException`|
|Обязательна обработка|Да|Нет|
|Примеры|`IOException`, `SQLException`|`NullPointerException`, `IllegalArgumentException`|

В Java компилятор заставляет обработать checked exception через try-catch или `throws`.

**В Kotlin** — все исключения unchecked. Checked exceptions считаются антипаттерном (Kotlin дизайн-решение) так как загрязняют сигнатуры и провоцируют пустые catch-блоки.

### ❓ Какую архитектуру использовал в проекте?

**Ответ:** MVVM + элементы Clean Architecture:

- **UI Layer** — Fragment, RecyclerView. Подписан на StateFlow из ViewModel через `repeatOnLifecycle(STARTED)`

### ❓ Автобоксинг и Анбоксинг — что это и какие проблемы?

**Ответ:** Автобоксинг — автоматическое преобразование примитива в обёртку, анбоксинг — обратно:

```java
// Автобоксинг: int → Integer
Integer x = 5;           // компилятор: Integer.valueOf(5)

// Анбоксинг: Integer → int
int y = x;               // компилятор: x.intValue()

// В коллекции — только объекты, поэтому boxing неизбежен
List<Integer> list = new ArrayList<>();
list.add(42);            // boxing
int val = list.get(0);   // unboxing
```

**Проблемы:**

```java
// 1. NullPointerException при анбоксинге null
Integer a = null;
int b = a;               // NPE!

// 2. Ловушка с кэшем Integer (-128..127)
Integer i1 = 127; Integer i2 = 127;
System.out.println(i1 == i2);   // true (кэш)

Integer i3 = 128; Integer i4 = 128;
System.out.println(i3 == i4);   // false (разные объекты!)
System.out.println(i3.equals(i4)); // true — правильно

// 3. Производительность — boxing в цикле создаёт много мусора для GC
long sum = 0;
for (Integer i : bigList) { sum += i; } // unboxing на каждой итерации
```

**В Kotlin:** `Int` компилируется в примитив `int` везде где возможно. В коллекциях (`List<Int>`) — boxing в `Integer`. Для массивов без boxing: `IntArray` вместо `Array<Int>`.

### ❓ Архитектура JVM — как устроена?

**Ответ:**

```
JVM = ClassLoader + Runtime Memory Areas + Execution Engine

Runtime Memory Areas:
┌─────────────────────────────────────────┐
│  Heap (общая для всех потоков)          │
│  ├── Young Generation (Eden, S0, S1)    │
│  └── Old Generation (Tenured)           │
├─────────────────────────────────────────┤
│  Metaspace (классы, методы — вне Heap)  │
├─────────────────────────────────────────┤
│  Stack (у каждого потока свой)          │
│  ├── Frame для каждого вызова метода    │
│  └── Локальные переменные, операнды    │
├─────────────────────────────────────────┤
│  PC Register  │  Native Method Stack    │
└─────────────────────────────────────────┘
```

**GC (Garbage Collector):**

- Объекты создаются в Eden (Young Gen)
- После нескольких GC-циклов живые объекты → Old Gen
- Minor GC — чистит Young Gen (быстро)
- Major/Full GC — чистит всё (долго, может вызвать паузу)

**В Android — ART вместо JVM:**

- AOT (Ahead-Of-Time) компиляция при установке — байткод → нативный код
- Быстрее запуск и выполнение, но больше места
- До Android 5.0 был Dalvik (JIT-компиляция)

### ❓ Локальные и Анонимные классы — в чём разница?

**Ответ:**

**Локальный класс** — объявлен внутри метода:

```java
void processContacts() {
    // Локальный класс — имеет имя, но виден только внутри метода
    class ContactFilter {
        boolean isValid(String phone) { return phone.startsWith("+"); }
    }
    ContactFilter filter = new ContactFilter();
}
```

**Анонимный класс** — без имени, сразу реализует интерфейс или наследует класс:

```java
// Старый стиль — сейчас заменяется лямбдами
button.setOnClickListener(new View.OnClickListener() {
    @Override
    public void onClick(View v) { loadContacts(); }
});

// С лямбдой (функциональный интерфейс):
button.setOnClickListener(v -> loadContacts());
```

**Важно:** анонимный/локальный класс захватывает переменные из внешнего scope. Они должны быть `effectively final` — иначе компилятор ругается. Это причина почему в лямбдах нельзя изменять внешние переменные.

### ❓ Обёртки для примитивов — зачем нужны?

**Ответ:** Java — не чисто объектный язык. Примитивы (`int`, `boolean`) не объекты. Обёртки нужны когда требуется объект:

```java
// 1. Коллекции принимают только объекты
List<Integer> list = new ArrayList<>();  // не List<int>!

// 2. Полезные методы
Integer.parseInt("42");
Integer.toBinaryString(255);
Integer.MAX_VALUE;           // 2147483647
Double.isNaN(0.0 / 0.0);

// 3. Nullable значения
Integer nullableAge = null;  // int не может быть null

// 4. Generics — только ссылочные типы
Optional<Integer> opt = Optional.of(42);
```

Все обёртки: `Byte`, `Short`, `Integer`, `Long`, `Float`, `Double`, `Character`, `Boolean`. Все иммутабельны, наследуют `Number` (кроме `Boolean`, `Character`).

### ❓ Примитивные и ссылочные типы — в чём разница?

**Ответ:**

||Примитивные|Ссылочные|
|---|---|---|
|Примеры|`int`, `long`, `boolean`, `char`|`String`, `Integer`, объекты|
|Хранение|На стеке|Объект в Heap, ссылка на стеке|
|Сравнение|`==` сравнивает значение|`==` сравнивает ссылку, `equals()` — значение|
|Default|`0`, `false`, `'\0'`|`null`|
|Nullable|Нет|Да|
|Передача|По значению (копия)|Ссылка по значению (копия ссылки)|

```java
// Передача примитива — изменение внутри метода не влияет на оригинал
void change(int x) { x = 100; }
int a = 5; change(a); // a == 5

// Передача объекта — изменение полей видно снаружи
void change(Contact c) { c.name = "New"; }
Contact c = new Contact("Old"); change(c); // c.name == "New"

// НО: переприсвоение ссылки — не влияет на оригинал
void change(Contact c) { c = new Contact("Another"); }
Contact c = new Contact("Old"); change(c); // c.name == "Old"
```

### ❓ Стеки и очереди в Java — какие реализации?

**Ответ:**

**Stack (LIFO — последний вошёл, первый вышел):**

```java
// Устаревший java.util.Stack (synchronized, медленный)
Stack<Integer> stack = new Stack<>();
stack.push(1); stack.push(2);
stack.pop();   // 2
stack.peek();  // смотрит без удаления

// Современный вариант — ArrayDeque как стек
Deque<Integer> stack = new ArrayDeque<>();
stack.push(1); stack.push(2);
stack.pop();   // 2
```

**Queue (FIFO — первый вошёл, первый вышел):**

```java
// LinkedList как очередь
Queue<String> queue = new LinkedList<>();
queue.offer("a"); queue.offer("b");
queue.poll();    // "a" — забирает и удаляет
queue.peek();    // "b" — смотрит без удаления

// ArrayDeque — быстрее LinkedList
Queue<String> queue = new ArrayDeque<>();

// PriorityQueue — с приоритетом (min-heap по умолчанию)
PriorityQueue<Integer> pq = new PriorityQueue<>();
pq.offer(3); pq.offer(1); pq.offer(2);
pq.poll(); // 1 (наименьший)
```

**Deque** — двусторонняя очередь, работает и как стек и как очередь.

### ❓ abstract class vs interface — когда что?

**Ответ:**

||Abstract class|Interface|
|---|---|---|
|Наследование|Только одно|Множественная реализация|
|Поля|Любые (state)|Только `public static final`|
|Конструктор|Есть|Нет|
|Методы|Любые|`default` методы (Java 8+)|
|Использование|Общая база с состоянием|Контракт поведения|

```java
// Abstract class — частичная реализация, есть общее состояние
abstract class BaseRepository {
    protected final Context context;
    BaseRepository(Context ctx) { this.context = ctx; }
    abstract List<Contact> getAll();
    protected void logError(String msg) { Log.e(TAG, msg); } // общий код
}

// Interface — чистый контракт
interface ContactsApi {
    List<Contact> getAll();
    default void refresh() { getAll(); } // Java 8+ default
}

// В Kotlin: interface может иметь реализации методов
interface Clickable {
    fun click()
    fun showOff() = println("I'm clickable") // реализация по умолчанию
}
```

**Правило:** "is-a" (является) → abstract class. "can-do" (умеет) → interface.

### ❓ ArrayList, LinkedList, HashMap, HashSet — разница и сложность операций

**Ответ:**

**ArrayList vs LinkedList:**

```
                ArrayList    LinkedList
get(i)          O(1)         O(n)
add(конец)      O(1)*        O(1)
add(середина)   O(n)         O(n) — найти + O(1) вставить
remove(середина)O(n)         O(n) — найти + O(1) удалить
память          компактная   много (узел + 2 ссылки)
```

`*` — амортизированное. При переполнении capacity → resize = O(n)

**ArrayList** — почти всегда быстрее. `LinkedList` полезен только если часто вставляешь/удаляешь с **известным** итератором.

**HashMap:**

- Внутри: массив + linked list/tree в каждом bucket
- `get`/`put`/`remove` — O(1) среднее, O(n) худший (все в одном bucket)
- Java 8+: при > 8 элементов в bucket → TreeMap (O(log n))
- Не гарантирует порядок. `LinkedHashMap` — сохраняет порядок вставки. `TreeMap` — сортированный по ключу

**HashSet:**

- Внутри — `HashMap<E, PRESENT>`. Все операции O(1)
- `TreeSet` — сортированный, O(log n) операции
- `LinkedHashSet` — сохраняет порядок вставки

### ❓ Concurrent collections — что и когда использовать?

**Ответ:**

```java
// ConcurrentHashMap — потокобезопасный HashMap
// Java 8: CAS + синхронизация на bucket уровне (не на всей карте)
ConcurrentHashMap<String, Contact> map = new ConcurrentHashMap<>();
map.put("key", contact);
map.computeIfAbsent("key", k -> loadContact(k)); // атомарно

// CopyOnWriteArrayList — при каждом write создаёт копию массива
// Хорошо: много чтений, редко запись
// Плохо: частые записи — дорого
CopyOnWriteArrayList<Contact> list = new CopyOnWriteArrayList<>();

// BlockingQueue — для producer-consumer паттерна
BlockingQueue<Contact> queue = new LinkedBlockingQueue<>(100);
queue.put(contact);   // блокирует если полная
queue.take();         // блокирует если пустая

// AtomicInteger/Long — атомарные операции без synchronized
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet(); // атомарный i++
counter.compareAndSet(5, 10); // CAS — меняет 5 на 10 только если значение == 5
```

В Kotlin корутинах вместо concurrent collections часто используют `Mutex` или `Channel` — избегают shared mutable state.

### ❓ final в Java — что означает для класса, метода, поля?

**Ответ:**

```java
// final класс — нельзя наследоваться
final class ImmutableContact { ... }
// class Sub extends ImmutableContact { } // ошибка

// final метод — нельзя переопределить в наследнике
class Base {
    final void doWork() { }
}

// final поле — можно присвоить только один раз
class Config {
    final String url;
    Config(String url) { this.url = url; } // OK — в конструкторе
    // url = "other"; // ошибка после инициализации
}

// final + static = константа (по соглашению UPPER_CASE)
static final int MAX_CONTACTS = 1000;

// Effectively final — переменная не меняется, компилятор сам понимает
// (нужно для лямбд и анонимных классов)
String name = "Ivan"; // effectively final
button.setOnClickListener(v -> System.out.println(name)); // OK
```

**В Kotlin:** все классы и методы `final` по умолчанию. Нужно явно написать `open` чтобы разрешить наследование. `val` = effectively final.

### ❓ Iterable и Iterator — как работает for-each?

**Ответ:**

```java
// for-each компилируется в использование Iterator
for (Contact c : contacts) { process(c); }

// Это то же самое что:
Iterator<Contact> it = contacts.iterator();
while (it.hasNext()) {
    Contact c = it.next();
    process(c);
}

// Iterable — интерфейс с одним методом
interface Iterable<T> {
    Iterator<T> iterator();
}

// Iterator — интерфейс обхода
interface Iterator<T> {
    boolean hasNext();
    T next();
    default void remove() { throw new UnsupportedOperationException(); }
}
```

**Своя реализация в Kotlin:**

```kotlin
class NumberRange(val from: Int, val to: Int) : Iterable<Int> {
    override fun iterator() = object : Iterator<Int> {
        var current = from
        override fun hasNext() = current <= to
        override fun next() = current++
    }
}

for (n in NumberRange(1, 5)) { println(n) } // 1 2 3 4 5
```

**ConcurrentModificationException** — если изменять коллекцию во время итерации. Используй `it.remove()` или `removeIf()`.

### ❓ Java 8 Streams — терминальные и промежуточные операции

**Ответ:**

```java
List<Contact> contacts = getContacts();

// Промежуточные (lazy, возвращают Stream) → терминальная (запускает всё)
List<String> result = contacts.stream()
    // --- Промежуточные ---
    .filter(c -> c.name != null)          // фильтрация
    .map(c -> c.name.toUpperCase())       // трансформация
    .sorted()                             // сортировка
    .distinct()                           // уникальные
    .limit(10)                            // первые N
    .skip(5)                              // пропустить N
    // --- Терминальная (запускает pipeline) ---
    .collect(Collectors.toList());

// Другие терминальные:
long count = contacts.stream().filter(...).count();
Optional<Contact> first = contacts.stream().findFirst();
boolean anyMatch = contacts.stream().anyMatch(c -> c.name.isEmpty());
String joined = contacts.stream().map(c -> c.name).collect(Collectors.joining(", "));
Map<String, List<Contact>> byCity = contacts.stream()
    .collect(Collectors.groupingBy(c -> c.city));
```

**Kotlin vs Java Streams:**

```kotlin
// Kotlin коллекции — eager (сразу вычисляются)
contacts.filter { it.name != null }.map { it.name!!.uppercase() }.take(10)

// Kotlin sequences — lazy (как Java Streams, лучше для больших данных)
contacts.asSequence()
    .filter { it.name != null }
    .map { it.name!!.uppercase() }
    .take(10)
    .toList() // терминальная операция
```

### ❓ JVM стек и байт-код — как работает выполнение?

**Ответ:**

**JVM Stack:**

- У каждого потока свой стек
- При вызове метода создаётся Stack Frame
- Frame содержит: локальные переменные, операндный стек, ссылку на константный пул
- При выходе из метода — frame удаляется
- `StackOverflowError` — слишком глубокая рекурсия, стек переполнен

**Байт-код:**

```java
// Java код:
int add(int a, int b) { return a + b; }

// Байт-код (iload = load int, iadd = add ints, ireturn = return int):
// iload_1   // загрузить a на операндный стек
// iload_2   // загрузить b
// iadd      // сложить (pop 2, push результат)
// ireturn   // вернуть
```

**INVOKEVIRTUAL** — байт-код инструкция для вызова виртуальных методов (полиморфизм):

```
INVOKEVIRTUAL — вызов метода через vtable (виртуальная таблица)
INVOKESTATIC  — статический метод
INVOKEINTERFACE — метод интерфейса
INVOKESPECIAL — private методы, конструкторы, super
INVOKEDYNAMIC — лямбды (Java 8+)
```

### ❓ HashMap vs TreeMap и HashSet vs TreeSet

**Ответ:**

```java
// HashMap — O(1) средне, не упорядочен
Map<String, Integer> hashMap = new HashMap<>();

// TreeMap — O(log n), отсортирован по ключу
Map<String, Integer> treeMap = new TreeMap<>();
treeMap.put("banana", 2); treeMap.put("apple", 1);
treeMap.firstKey(); // "apple" — минимальный
treeMap.subMap("a", "c"); // все ключи от "a" до "c"

// LinkedHashMap — O(1), сохраняет порядок вставки (полезно для LRU Cache)
Map<String, Integer> linkedMap = new LinkedHashMap<>();
```

```java
// HashSet — O(1), не упорядочен
Set<String> hashSet = new HashSet<>();

// TreeSet — O(log n), отсортирован (использует TreeMap внутри)
TreeSet<Integer> treeSet = new TreeSet<>();
treeSet.add(3); treeSet.add(1); treeSet.add(2);
treeSet.first();   // 1
treeSet.last();    // 3
treeSet.floor(2);  // наибольший <= 2
treeSet.ceiling(2);// наименьший >= 2

// LinkedHashSet — O(1), сохраняет порядок вставки
```

**Когда TreeMap/TreeSet:** нужна сортировка или range-запросы (от A до B).

### ❓ List в Java — какие реализации и когда?

**Ответ:**

```java
// ArrayList — индексированный массив, O(1) random access
List<Contact> list = new ArrayList<>();         // дефолтный выбор
List<Contact> list = new ArrayList<>(1000);    // с начальной capacity

// LinkedList — двусвязный список
// Используй ТОЛЬКО если: частые вставки/удаления с известным итератором
// В реальности ArrayDeque быстрее для очереди/стека
List<Contact> linked = new LinkedList<>();

// Arrays.asList — фиксированный размер (нельзя add/remove!)
List<String> fixed = Arrays.asList("a", "b"); // fixed size
List<String> mutable = new ArrayList<>(Arrays.asList("a", "b")); // mutable

// List.of (Java 9+) — immutable
List<String> immutable = List.of("a", "b", "c");

// В Kotlin:
val immutable = listOf("a", "b")        // List — только чтение
```

### ❓ Коллекции в Kotlin — List/MutableList, Map, Set и их особенности

**Ответ:**

В Kotlin коллекции разделены на read-only и mutable на уровне интерфейсов:

```kotlin
// Read-only (не означает immutable — underlying может быть MutableList!)
val list: List<Int> = listOf(1, 2, 3)
val map: Map<String, Int> = mapOf("a" to 1, "b" to 2)
val set: Set<Int> = setOf(1, 2, 3)

// Mutable
val mList: MutableList<Int> = mutableListOf(1, 2, 3)
val mMap: MutableMap<String, Int> = mutableMapOf("a" to 1)
val mSet: MutableSet<Int> = mutableSetOf(1, 2, 3)

// Полезные операции:
list.filter { it > 1 }           // [2, 3]
list.map { it * 2 }              // [2, 4, 6]
list.groupBy { it % 2 == 0 }     // {false=[1,3], true=[2]}
list.partition { it > 1 }        // Pair([2,3], [1])
list.zip(listOf("a","b","c"))    // [(1,a), (2,b), (3,c)]
list.flatten()                    // для List<List<T>>
list.flatMap { listOf(it, it*2)} // [1,2, 2,4, 3,6]
list.associate { it to it*2 }   // {1=2, 2=4, 3=6}

// Sequences — ленивые вычисления (для больших данных)
list.asSequence().filter { it > 1 }.map { it * 2 }.toList()
```

### ❓ XML vs Compose — главные отличия

**Ответ:**

||XML (View)|Jetpack Compose|
|---|---|---|
|Парадигма|Императивная — как изменить|Декларативная — что показать|
|Обновление UI|`view.text = "..."` вручную|Рекомпозиция при изменении State|
|Иерархия|View tree (измерение → layout → draw)|Composable функции|
|RecyclerView|Adapter + ViewHolder|`LazyColumn` / `LazyRow`|
|Анимации|Animator, TransitionManager|`animate*AsState`, `AnimatedVisibility`|
|Тестирование|Espresso|Compose UI Test (`composeRule`)|

```kotlin
// XML подход
binding.nameText.text = user.name  // императивно

// Compose подход
@Composable
fun UserCard(user: User) {
    Text(text = user.name)  // декларативно — перерисуется автоматически при изменении user
}
```

**Когда XML:** легаси проект, нужна обратная совместимость с API < 21 в деталях. **Когда Compose:** новый проект, Google рекомендует как стандарт с 2021.

### ❓ Android KTX — что даёт и зачем?

**Ответ:** Android KTX — набор Kotlin extension functions для Android API. Делает код лаконичнее без новой функциональности.

```kotlin
// Без KTX:
sharedPreferences.edit().putString("key", "value").apply()
// С KTX:
sharedPreferences.edit { putString("key", "value") }

// Без KTX:
view.viewTreeObserver.addOnGlobalLayoutListener(object : ViewTreeObserver.OnGlobalLayoutListener {
    override fun onGlobalLayout() {
        view.viewTreeObserver.removeOnGlobalLayoutListener(this)
        doSomething()
    }
})
// С KTX:
view.doOnLayout { doSomething() }

// Другие примеры:
fragment.viewLifecycleOwner.lifecycleScope  // lifecycle-ktx
uri.toFile()                                // core-ktx
bitmap.scale(width, height)                // core-ktx
context.getColor(R.color.primary)          // core-ktx
bundle { putString("key", "value") }       // core-ktx
```

Основные артефакты: `core-ktx`, `lifecycle-ktx`, `fragment-ktx`, `navigation-ktx`, `room-ktx`.

### ❓ Тестирование — Unit vs Integration vs UI тесты

**Ответ:**

**Unit тесты** — самые быстрые, без Android:

```kotlin
// JUnit4 + MockK
class ContactsViewModelTest {
    @get:Rule val mainDispatcherRule = MainDispatcherRule() // тест-диспатчер

    private val repository = mockk<ContactsRepository>()
    private lateinit var viewModel: ContactsViewModel

    @Before fun setup() { viewModel = ContactsViewModel(repository) }

    @Test fun `load contacts success`() = runTest {
        val fakeContacts = listOf(Contact(1, "Ivan", "+79991234567"))
        coEvery { repository.getContacts() } returns fakeContacts

        viewModel.loadContacts()

        assertEquals(UiState.Success(fakeContacts), viewModel.uiState.value)
        coVerify { repository.getContacts() }
    }
}
```

**Integration тесты** — несколько слоёв, может использовать Room in-memory:

```kotlin
// Room in-memory database — быстро, не нужен эмулятор
val db = Room.inMemoryDatabaseBuilder(context, AppDatabase::class.java).build()
val dao = db.contactDao()

runTest {
    dao.insert(Contact(1, "Ivan", "+7999"))
    val result = dao.getAll().first()
    assertEquals(1, result.size)
}
```

**UI тесты (Espresso/Compose)** — нужен эмулятор/устройство:

```kotlin
// Compose UI Test
@get:Rule val composeRule = createComposeRule()

@Test fun contacts_displayed() {
    composeRule.setContent { ContactsScreen(fakeViewModel) }
    composeRule.onNodeWithText("Ivan").assertIsDisplayed()
    composeRule.onNodeWithTag("delete_button").performClick()
}
```

**Пирамида тестов:** много Unit → меньше Integration → мало UI (дорогие и медленные).
